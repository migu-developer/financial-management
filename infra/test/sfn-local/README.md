# ChatProcess state-machine testing (no deploy)

Two complementary ways to test the `ChatProcess` Step Functions workflow
(`infra/lib/versions/v2/step-functions-chat-stack.ts`) without deploying to a
live account:

| Tool                        | What it tests                                                               | Where it runs     |
| --------------------------- | --------------------------------------------------------------------------- | ----------------- |
| **D3** Step Functions Local | Whole-execution branching, retries and catches with **mocked** task results | Local Docker / CI |
| **D4** TestState API        | A **single** state's routing (e.g. the `Intent?` Choice) against live DEV   | DEV account only  |

---

## D3 — Step Functions Local + MockConfigFile

[Step Functions Local](https://docs.aws.amazon.com/step-functions/latest/dg/sfn-local.html)
runs the real state-machine interpreter in a Docker container. A
[MockConfigFile](https://docs.aws.amazon.com/step-functions/latest/dg/sfn-local-test-sm-exec.html)
replaces every `Task` state's integration call with a canned response, so we
can drive every branch — including the failure and retry paths — deterministically
and offline.

### Files

- `mock-config.json` — the MockConfigFile. Top-level `StateMachines` →
  `ChatProcessTest` → `TestCases` (state name → mocked-response name) and
  `MockedResponses` (`Return` / `Throw` payloads, keyed by retry attempt index
  e.g. `"0"`, or a range like `"0-2"`).
- `extract-asl.mjs` — reads the synthesized CloudFormation template and writes
  the resolved ASL to `chat-process.asl.json`. The CDK emits the
  `DefinitionString` as an `Fn::Join` over literals + intrinsics
  (`Fn::GetAtt` task ARNs, `Ref` partition/region); this resolves each to a
  placeholder ARN so the local engine can load it. The mock config matches on
  **state name**, so placeholder ARNs are fine. It **also rewrites every Bedrock
  task** (`arn:aws:states:::bedrock:invokeModel`) into a mockable
  `arn:aws:states:::lambda:invoke` task — see the caveat below.
- `chat-process.asl.json` — generated; the resolved ASL (gitignored / a build
  artifact, regenerated by the runner).
- `assert-coverage.mjs` — **completeness guard**: fails if any `Task` state in
  the ASL isn't referenced by at least one `TestCase` (or if a test case points
  at a stale state / undefined mocked response). The runner invokes it right
  after extracting the ASL — so adding a new state to the state machine without
  a matching test case **breaks CI** instead of going silently untested.
- `../../scripts/sfn-local-test.sh` — the runner.

### Mocked Bedrock response shapes

The two model families return different bodies, and the stack's
`resultSelector` reads different paths, so the mocks must match:

- **Nova** (Micro/Lite — intent, SQL params, field extraction):
  `$.Body.output.message.content[0].text` holds a string (the intent token or a
  JSON string).
- **Claude** (Haiku — every NL response): `$.Body.content[0].text`.

### Caveat — Bedrock tasks are rewritten to `lambda:invoke` locally

Step Functions Local (2.x) **cannot execute the Bedrock optimized integration**
(`arn:aws:states:::bedrock:invokeModel`) — `StartExecution` returns HTTP 500 even
when the task is mocked. So `extract-asl.mjs` rewrites each Bedrock task in the
**extracted (local) ASL only** to a `lambda:invoke` task with valid
`Parameters` (`FunctionName` + `Payload.$: "$"`). The mock matches on state name
and returns a Bedrock-shaped `Body`, so each task's `ResultSelector`
(`$.Body.output…` / `$.Body.content…`) still resolves, and the `Retry`/`Catch`
on `Bedrock.*` errors are preserved (the mock can still `Throw` them).

What this means: D3 validates the **control flow** (every branch, every
retry/catch, the choices) faithfully, but it does **not** exercise the real
Bedrock integration's request shaping (the `States.Format` prompt building). That
is covered by the `@packages/prompts` unit tests, the CDK assertions, and the
live smoke test (D5). The **deployed** ASL is never modified.

### Test cases covered

| Test case                               | Path exercised                                               | Expected status |
| --------------------------------------- | ------------------------------------------------------------ | --------------- |
| `queryHappyPath`                        | QUERY branch end-to-end                                      | `SUCCEEDED`     |
| `createCompleteConfirmed`               | CREATE → complete → preview → **confirmed** → create expense | `SUCCEEDED`     |
| `createCompleteCancelled`               | CREATE → complete → preview → **cancelled**                  | `SUCCEEDED`     |
| `createCompleteSuperseded`              | CREATE → preview → **superseded** (`PreviewSuperseded`)      | `SUCCEEDED`     |
| `createCompleteExpired`                 | CREATE → preview → **`States.Timeout`** → `PreviewExpired`   | `SUCCEEDED`     |
| `createIncompleteClarification`         | CREATE → **incomplete** → clarification branch               | `SUCCEEDED`     |
| `unknownIntent`                         | UNKNOWN branch                                               | `SUCCEEDED`     |
| `executeQueryFailsRoutesToPublishError` | `ExecuteQuery` throws → catch-all → `PublishError` → `Fail`  | `FAILED`        |
| `bedrockThrottleRecovers`               | `ClassifyIntent` throttles 3×, **Retry recovers** on 4th     | `SUCCEEDED`     |

The last two are the resilience cases: one proves a task failure routes to the
friendly-error path and still fails the execution (so `ExecutionsFailed`
alarms), the other proves the Bedrock `Retry` (maxAttempts 4) absorbs transient
`ThrottlingException`s.

### Run it locally

Prerequisites: Docker, AWS CLI v2, Node ≥ 18, pnpm.

```bash
# from the repo root
pnpm --filter @infra test:sfn-local
# or directly
infra/scripts/sfn-local-test.sh
```

The runner synthesizes the stack, extracts the ASL, boots
`amazon/aws-stepfunctions-local`, creates the state machine, runs every test
case against the `arn:...:stateMachine:ChatProcessTest#<TestCaseName>` qualified
ARN, asserts each final status, and exits non-zero on any mismatch.

> The credentials are dummy locals — Step Functions Local never calls AWS.

### In CI

This suite runs as the **`sfn-local` job in `.github/workflows/ci.yml`** (the
quality workflow), on every PR and on push to `main`, in parallel with the
lint/typecheck/test job. `ubuntu-latest` ships Docker and the AWS CLI, so the
container runs natively.

### Adding a new state to the workflow

When you add a `Task` state to `step-functions-chat-stack.ts`:

1. Add a `MockedResponse` for it in `mock-config.json` (`Return` or `Throw`).
2. Reference it from the relevant `TestCase`(s) (and add a new TestCase +
   expected status in `sfn-local-test.sh` if it introduces a new branch).

If you skip step 1–2, the completeness guard fails the `sfn-local` CI job with
the exact uncovered state name — coverage of new states is **enforced**, not
hoped for.

## D4 — TestState API

`scripts/sfn-teststate.ts` uses the `TestState` API to test a single state's
routing in isolation against a **real DEV account**. It currently asserts the
`Intent?` Choice (`QUERY`/`CREATE`/gibberish → correct next state, including
trailing-whitespace tolerance) and the `Confirmed?` Choice
(superseded/confirmed/declined).

### Run it (DEV ONLY)

```bash
SFN_TESTSTATE_STAGE=dev \
SFN_TESTSTATE_ROLE_ARN=arn:aws:iam::<dev-account>:role/<teststate-role> \
AWS_REGION=us-east-1 \
pnpm --filter @infra test:sfn-teststate
```

The script **refuses to run** unless `SFN_TESTSTATE_STAGE=dev`. `TestState` can
invoke the live task resources a state references, so it must never be pointed
at prod.

### Narrow IAM role for TestState

The role passed in `SFN_TESTSTATE_ROLE_ARN` should be a dedicated, least-
privilege role assumable only in DEV:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "TestState",
      "Effect": "Allow",
      "Action": "states:TestState",
      "Resource": "*"
    },
    {
      "Sid": "TaskInvokePermsForTestedStates",
      "Effect": "Allow",
      "Action": ["bedrock:InvokeModel", "lambda:InvokeFunction"],
      "Resource": [
        "arn:aws:bedrock:us-east-1:<dev-account>:inference-profile/us.*",
        "arn:aws:bedrock:*::foundation-model/*",
        "arn:aws:lambda:us-east-1:<dev-account>:function:fm-dev-chat-*"
      ]
    }
  ]
}
```

- The Choice states tested here invoke **no** task, so `states:TestState` alone
  is sufficient for them. The task-invoke statement is only needed if you extend
  the script to test `Task` states (e.g. running `ClassifyIntent` against the
  live Bedrock profile).
- Scope `lambda:InvokeFunction` to `fm-dev-chat-*` and Bedrock to the DEV
  account / profiles — never include prod ARNs.
